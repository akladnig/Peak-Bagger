## Problem

`My Peak Lists` and `Map` currently cannot sort or filter peaks by the metadata the user now relies on for planning and comparison. `Peak` already stores numeric `rating` and free-text `difficulty`, but `My Peak Lists` does not expose them in its peak table, `Map` does not provide peak metadata filtering, and `Peak` has no persisted duration data at all. The new feature must add `Peak duration` as maintained metadata, preserve region-specific `Peak difficulty` semantics, and add user-facing sort/filter controls that fit the app's existing Flutter UI patterns. [L1] [L2] [L3] [L4] [L5] [L6] [L7] [L8] [L9] [L10] [L11]

## Proposed Outcome

`Peak` stores both a numeric and display-label form of `Peak duration`, `My Peak Lists` exposes `Rating`, `Difficulty`, and `Duration` as additional sortable peak-table columns, and `Map` gains one visible app-bar `Filter` control to the right of search that opens a fixed three-row popup for `Rating`, `Difficulty`, and `Duration`. The feature keeps `Peak difficulty` region-specific, renders `Rating` as nearest-half stars while preserving numeric sort/filter behavior, keeps map filters active within the same app session, and uses existing admin/import maintenance paths so the new duration metadata can be populated and verified deterministically. [L1] [L2] [L3] [L4] [L5] [L6] [L7] [L8] [L9] [L10] [L11]

## User Stories

1. As a `My Peak Lists` user, I can sort the peak-details table by `Rating`, `Difficulty`, and `Duration` in addition to the existing columns so I can compare peaks by quality, grading, and expected commitment. [L4] [L6]
2. As a Map user, I can open one `Filter` popup and narrow visible peaks by `Rating`, `Difficulty`, and `Duration` without cluttering the action rail with multiple separate controls. [L5] [L6] [L8] [L10] [L11]
3. As a user browsing mixed-region peak sets, I get exact, region-aware `Peak difficulty` filtering and predictable cross-region sorting without the app pretending all grading systems share one global ladder. [L1] [L3] [L9]
4. As a maintainer or importer, I can populate and correct `Peak duration` through the same app-owned maintenance paths already used for peak metadata, and invalid duration input fails clearly instead of silently creating unusable filter data. [L2] [L7]

## Requirements

1. Use `Peak difficulty` and `Peak duration` as the canonical project terms for this feature. `Peak difficulty` refers to the region-specific grade stored on `Peak` and must remain distinct from `Hiking difficulty`. [L1] [L2]
2. Add persisted `Peak duration` fields to `Peak` for both machine behavior and user-facing display. The source-of-truth numeric field is `durationMinutes`, and the display field is `durationLabel`. `Peak duration` represents an estimated standard out-and-back walking duration for one peak. [L2]
3. `Peak duration` parsing must accept only these input families: exact clock format `H:MM` such as `0:30` and `4:15`; hour ranges in the form `<int>-<int> hour` or `<int>-<int> hours` such as `4-5 hours`; and day ranges in the form `<int>-<int> day` or `<int>-<int> days` such as `2-3 days`. For ranges, the numeric `durationMinutes` value must use the upper bound. The supported duration domain for this feature must cover at least 15 minutes through about 20 days. [L2]
4. `My Peak Lists` is sort-only in this iteration. Do not add peak metadata filtering controls to the `My Peak Lists` screen. [L4]
5. The `My Peak Lists` peak-details table must show columns in this exact order: `Rating`, `Peak Name`, `Height`, `Ascent Date`, `Ascents`, `Difficulty`, `Duration`. [L4]
6. `My Peak Lists` must allow `Rating`, `Difficulty`, and `Duration` to sort using the same tap-to-toggle pattern already used by the other sortable columns. [L4]
7. The `Rating` table column must render a five-star row using full, half, and empty stars. It must not show decimal text in the same cell. When a peak has no rating, the cell is blank. Star rendering must round the stored numeric rating to the nearest half-star for display only. Sorting must use the existing numeric 0-5 rating value, not the rendered star glyphs. [L6]
8. The `Difficulty` table column must show the stored `Peak difficulty` text. Sorting must use these app-owned region ladders when applicable: Tasmania uses `Easy < Medium < Hard < Very Hard`; Italy administrative regions use `T < E < EE < EEA < EAI`; `slovenia` and `croatia` use `T1 < T2 < T3 < T4 < T5 < T6`. For regions without a configured ladder, sort alphabetically by stored difficulty text. If one visible table contains multiple regions, sort by region first, then by that region's difficulty order, then by peak name. [L3]
9. The `Duration` table column must show `durationLabel` when it is present. If no label is present but a numeric duration exists, show a formatted duration string derived from `durationMinutes`. Sorting must use `durationMinutes`. [L2]
10. For the new `Rating`, `Difficulty`, and `Duration` columns in `My Peak Lists`, rows with missing values must render blank and sort after rows with values, following the screen's existing blank-value sort conventions. [L4] [L6]
11. `Map` is filter-only for this feature. Use one visible control with the standard filter icon followed by a text label. Place it in the map app bar immediately to the right of the search control, add a vertical divider immediately after it, and move the search control from the centered app-bar position to the left side of the app bar. Do not add three always-visible filter FABs. [L8] [L11]
12. When no map metadata filters are active, the control text must be exactly `Filter` and use the unselected visual state. When one or more filters are active, the control text must be exactly `1 Filter`, `2 Filters`, or `3 Filters` based on the number of non-`Any` selections, and it must use the selected visual state. [L11]
13. Tapping the map `Filter` control must open one popup or modal filter surface that contains exactly three fixed filter rows in this order: `Rating`, `Difficulty`, `Duration`. Use the provided mockup only as a styling reference: each row uses a dark rounded container with a trailing dropdown styled like the mockup's rightmost control. Keep `Clear filters`. Do not add `Saved filters`, boolean operators, nested filters, per-row delete icons, or dynamic add/remove rows in this feature. [L10]
14. The map filter panel must apply option changes immediately. Closing the panel by outside tap, Escape, Back, or equivalent dismiss behavior must keep the current selection. `Clear filters` must reset all three rows to `Any` and keep the panel open. The map must update live so only matching peaks remain visible. [L8]
15. Map filter state must persist while the app stays open. Changing peak-list selection, changing visible regions, or leaving and returning to `Map` during the same app session must preserve the current `Rating`, `Difficulty`, and `Duration` selections. Do not silently clear active filters just because the current visible peaks no longer offer matching values. In that case, keep the filter active and show zero matches until the user changes or clears it. [L8]
16. The map `Rating` filter must be single-select with options `Any`, `3.0`, `3.5`, `4.0`, and `4.5`, presented with star-based visuals rather than plain decimal text. Matching behavior is `peak.rating >= selectedThreshold`. Peaks with missing ratings must be excluded whenever the selection is not `Any`. Accessibility semantics for displayed ratings must announce the numeric value, for example `4.5 out of 5 stars`. [L6]
17. The map `Difficulty` filter must be single-select with an `Any` option plus region-specific difficulty values drawn from the current visible/selected peak scope before applying map metadata filters. When multiple grading systems are present, group options by region and match one exact `(region, difficulty)` pair at a time. When needed for clarity, show region context in the selected value, such as `T4 (Slovenia)`. If the currently selected `(region, difficulty)` pair falls out of the current visible/selected scope, keep that selected value visible in the filter row so the user can still see and clear it while the normal option list reflects the current scope. Peaks with missing difficulty must be excluded whenever the selection is not `Any`. [L3] [L9]
18. The map `Duration` filter must be single-select with exactly these visible options: `Any`, `4h`, `8h`, `12h`, `2d`, `5d`, `10d`, `2d+`. `4h`, `8h`, `12h`, `2d`, `5d`, and `10d` match peaks where `durationMinutes <= threshold`. `2d+` matches peaks where `durationMinutes >= 2880`. Peaks with missing duration must be excluded whenever the selection is not `Any`. [L5]
19. Different regions may use different `Peak difficulty` systems. Keep exact-match region semantics for filtering and do not invent or expose a global normalized difficulty scale in this feature. [L1] [L3]
20. Add `Peak duration` maintenance support to existing app-owned metadata flows. The dedicated ObjectBox Admin peak details editor must let an admin edit the human-readable duration label and auto-derive `durationMinutes` from it. Both duration fields must remain inspectable through the generic admin/schema tooling. Ranked peak-list CSV import must accept an optional `duration` column and update both stored duration fields when present. Existing peaks with no duration remain valid and display blank until populated. [L7]
21. Invalid imported duration values must fail the import with clear row-level errors, matching the project's existing rating-import failure style. Do not silently coerce unsupported duration text into partial data. [L7]
22. This feature must not auto-derive `Peak duration` from ascent history, route timing, live ETA, external routing services, or other non-persisted runtime calculations. [L7]

## Technical Decisions

1. Extend the ObjectBox `Peak` model with two duration fields: a numeric `durationMinutes` source of truth for sorting and filtering, and a display-oriented `durationLabel` that preserves the imported or edited wording. Keep the existing numeric 0-5 `rating` field as the source of truth beneath the new star UI. [L2] [L6] [L7]
2. Implement duration parsing and formatting as shared app-owned logic so ranked CSV import, admin editing, table display, and map filter matching all follow the same supported formats and range-to-upper-bound rule. [L2] [L5] [L7]
3. Keep `Peak difficulty` ordering app-owned and region-aware. The ordering layer must accept region context, expose exact-match filter values per region, use these explicit ladders for this slice: Tasmania `Easy < Medium < Hard < Very Hard`, Italy administrative regions `T < E < EE < EEA < EAI`, and `slovenia`/`croatia` `T1 < T2 < T3 < T4 < T5 < T6`; and fall back to alphabetical ordering when a region has no explicit ladder yet. [L3] [L9]
4. Keep the new map metadata-filter state in existing map-owned Flutter state rather than introducing disk persistence for this slice. The state must survive route switches and list/region changes within the same app session but does not need a new app-restart persistence contract. [L8] [L11]
5. Reuse the existing app-bar search/control visual language for selected versus unselected control states where possible, while introducing a fixed three-row popup layout that follows the supplied mockup's visual styling without adopting its saved-filter or dynamic-builder features. The app bar must place search on the left, `Filter` immediately to its right, and a vertical divider immediately after the `Filter` control. [L10] [L11]
6. Any ObjectBox schema, admin row-mapping, or import contract change introduced by `Peak duration` must stay aligned across generated schema files, admin inspection/editing surfaces, and schema-guard regression coverage so the new fields remain inspectable and writable through the repository's existing tooling. [L7]

## Testing Strategy

1. Use behavior-first TDD for the new non-UI logic: duration parsing/formatting, duration filter matching, rating threshold matching, and region-aware difficulty ordering and filter grouping. Prefer focused unit or service-level tests for those rules before wiring them into widgets. [L2] [L3] [L5] [L6] [L9]
2. Extend peak metadata model and import/admin test coverage to prove the new duration fields persist and round-trip correctly. This includes targeted updates to `test/services/peak_list_import_service_test.dart`, `test/services/objectbox_admin_repository_test.dart`, `test/services/objectbox_schema_guard_test.dart`, and any directly affected peak model or admin-editor tests. Cover valid exact durations, valid ranges, blank values, and invalid duration failure messages. [L2] [L7]
3. Extend `test/widget/peak_lists_screen_test.dart` with deterministic widget coverage for the new table contract: exact column order, nearest-half star rendering in the `Rating` column, sortable `Rating`/`Difficulty`/`Duration` headers, numeric-versus-text sort behavior, configured difficulty-ladder ordering, formatted duration display, and blank-value ordering rules. [L3] [L4] [L6]
4. Extend map-focused widget coverage, likely in `test/widget/tasmap_map_screen_test.dart`, map app-bar tests, and related map selection tests, to cover the visible `Filter` control, the moved search placement, the trailing vertical divider, its `Filter` / `1 Filter` / `2 Filters` / `3 Filters` label transitions, selected-versus-unselected state, popup open/close behavior, immediate apply behavior, `Clear filters`, zero-match persistence, stale selected difficulty visibility, and session persistence across map route revisits and peak-list or visible-region changes. [L8] [L9] [L10] [L11]
5. Add deterministic widget coverage for the popup structure and option sets: exactly three fixed rows, dark rounded row containers, trailing dropdown affordances, star-based rating choices, grouped mixed-region difficulty options, retention of a stale selected difficulty value, and the exact duration option list `Any`, `4h`, `8h`, `12h`, `2d`, `5d`, `10d`, `2d+`. Add stable selectors for the filter trigger, popup root, each row, and `Clear filters` if existing keys are insufficient. [L5] [L6] [L9] [L10] [L11]
6. Prefer existing fake repositories, provider overrides, and deterministic in-memory map data. Automated coverage for this feature must not require live network calls, live map services, or secrets. Widget and unit/service coverage should be sufficient unless implementation reveals a gap that existing Flutter test seams cannot cover reliably. [L7] [L8]

## Out of Scope

1. Adding peak metadata filtering controls to `My Peak Lists`. [L4]
2. Introducing a global normalized `Peak difficulty` scale across regions. [L1] [L3]
3. Persisting the new map metadata filters across full app restarts. [L8]
4. Adding saved filters, boolean filter operators, nested filters, row deletion, or dynamically added filter rows to the map popup. [L10]
5. Auto-deriving `Peak duration` from tracks, route timing, or external services. [L7]
